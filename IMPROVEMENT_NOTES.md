# Steam Account Manager 改进建议

本文档记录当前项目中可以改进的风险点和优化方向，按优先级分组，便于后续逐项选择和实现。

## 高优先级

### ~~1. 明文保存账号密码~~

处理状态：暂不加密，已通过 `.gitignore` 降低误提交风险。

当前 `data/accounts.json` 中保存了 Steam 密码、邮箱、手机号、5E 信息等敏感数据，且没有加密。

风险：

- 项目目录被同步、备份、打包或误提交时，账号信息会直接暴露。
- 本机其他程序或用户只要能读取该文件，就能获取账号密码。

建议：

- 至少新增 `.gitignore`，忽略 `data/accounts.json`、`data/*.invalid-*.json`、`__pycache__/`、`.idea/` 等文件。
- 进一步可以用 Windows DPAPI 加密密码字段。
- 如果需要跨机器迁移，可以考虑“主密码 + 本地加密文件”的方案。

### 2. `steam.exe -login` 会暴露命令行参数

当前登录流程通过 `subprocess.Popen([steam_path, "-login", login_name, password])` 启动 Steam。

风险：

- Windows 进程命令行可能被本机其他进程、调试工具、WMI、日志或崩溃诊断看到。
- 密码作为命令行参数传递，本身不是安全做法。

建议：

- 如果继续使用 `-login`，在 UI 或文档中明确提示该风险。
- 后续调研是否存在更安全的 Steam 登录方式。
- 如果没有更安全入口，需要在便利性和安全性之间明确取舍。

### ~~3. 允许选择任意可执行文件作为 Steam~~

处理状态：已限制为只能保存和使用文件名为 `steam.exe` 的路径。

当前“选择 Steam 路径”允许选择 `*.exe` 或任意文件，后续只检查文件是否存在，没有校验它是否真的是 `steam.exe`。

风险：

- 用户误选恶意程序时，程序会把账号密码作为命令行参数传给它。
- `data/settings.json` 如果被篡改为恶意路径，也会造成账号密码泄露。

建议：

- 强制要求文件名为 `steam.exe`。
- 对非标准路径弹出强警告。
- 更进一步可以校验文件版本信息或数字签名发布者是否为 Valve。

### ~~4. JSON 保存不是原子写入~~

处理状态：已改为先写临时文件，再用 `os.replace()` 原子替换正式文件。

当前保存账号和设置时直接覆盖 `accounts.json` 或 `settings.json`。

风险：

- 写入过程中如果断电、程序崩溃、磁盘满或文件被占用，可能留下半截 JSON。
- 下次启动时数据文件会损坏。

建议：

- 保存时先写入临时文件，例如 `accounts.tmp.json`。
- 写入成功后用 `os.replace()` 原子替换正式文件。
- 可选：每次保存前保留一个 `.bak` 备份。

## 中优先级

### ~~5. 数据加载只处理 JSON 语法错误，不处理结构错误~~

处理状态：已校验 `accounts.json` 根节点必须是列表、列表项必须是对象；`settings.json` 根节点必须是对象。不符合时会备份原文件并按空数据继续启动。

当前只捕获 `json.JSONDecodeError`。如果文件是合法 JSON 但结构不符合预期，例如：

```json
{}
```

或：

```json
"abc"
```

程序仍可能在后续逻辑中崩溃。

风险：

- `accounts.json` 不是列表时，账号加载可能异常。
- `settings.json` 不是对象时，启动阶段调用 `.get()` 可能异常。

建议：

- 加载账号后校验根节点必须是 `list`。
- 列表项必须是 `dict`。
- 加载设置后校验根节点必须是 `dict`。
- 不合法时按损坏文件处理：备份原文件、提示用户、使用空数据继续启动。

### ~~6. 文件读取异常没有完整兜底~~

处理状态：已在加载账号和设置时捕获 `OSError`、`UnicodeDecodeError`、`JSONDecodeError` 和结构校验错误，并统一记录加载问题。

当前读取 JSON 时主要处理解析失败，没有完整处理：

- `OSError`
- `UnicodeDecodeError`
- 权限不足
- 文件被占用
- 编码损坏

风险：

- 数据文件无法读取时，程序可能直接启动失败。

建议：

- 仓库层统一捕获读取异常。
- 将错误记录为 `LoadIssue`。
- GUI 启动后弹窗提示具体文件和错误信息。

### 7. 登录成功状态不等于真实登录成功

当前只要 `subprocess.Popen` 成功，就会认为“已尝试启动登录流程”。

风险：

- Steam 启动成功不代表账号登录成功。
- Steam Guard、网络异常、客户端拒绝 `-login` 参数等情况无法被准确区分。

建议：

- UI 文案继续明确区分“已启动登录尝试”和“真实登录成功”。
- 后续可以调研检测 Steam 当前登录账号或客户端状态的方法。
- 如果无法准确检测，就不要自动更新为“登录成功”。

### ~~8. 自动强制关闭 Steam 过于激进~~

处理状态：已新增“Steam 退出方式”设置。默认先用 `steam.exe -shutdown` 请求正常退出，超时后询问是否强制结束；也可以在设置中选择直接强制结束。

当前检测到 `steam.exe` 或 `steamwebhelper.exe` 正在运行时，会自动调用 `taskkill /T /F` 强制结束。

风险：

- 可能中断 Steam 下载、更新、运行中的游戏或其他用户操作。
- 强杀进程可能导致 Steam 自身状态异常。

建议：

- 登录前弹出确认框：“检测到 Steam 正在运行，是否关闭并切换账号？”
- 增加设置项：自动关闭 / 每次询问 / 不关闭。
- 如果用户选择不关闭，则取消本次登录流程或提示手动处理。

### ~~9. 导入覆盖可能清空已有字段~~

处理状态：已调整覆盖规则。导入覆盖已有账号时，只用非空字段更新；空 `last_login` 和空备注不会再清掉已有数据。

导入解析默认会生成空 `last_login`，覆盖已有账号时会把已有 `last_login` 覆盖为空。

风险：

- 已有账号的运行状态或历史信息可能被导入流程意外清空。
- 备注字段也会被整体重写。

建议：

- 导入覆盖时只覆盖文本中明确解析到的字段。
- `last_login` 这类运行状态字段不应该由导入文本清空。
- 备注可以提供“覆盖备注”或“追加备注”的选项。

### ~~10. 批量导入分块逻辑较脆弱~~

处理状态：已增强分块逻辑。支持一行一个完整账号、一个账号拆成多行、以及单段文本按账号起始标记切分。

当前如果导入文本包含多行，逻辑会把每一行当成一个独立账号块。

风险：

- 很多真实文本是一组账号占多行，例如 Steam 账号一行、密码一行、邮箱一行。
- 这种格式容易解析失败或被跳过。

建议：

- 按账号起始标记、空行、`----` 等分隔。
- 或者做状态机解析，把连续多行合并成一个账号块。
- 为典型导入样例补测试，避免后续规则改坏。

## 低优先级 / 工程质量

### 11. 缺少自动化测试

当前没有测试文件。

建议优先补这些测试：

- `text_importer.py` 的导入样例测试。
- `repositories.py` 的损坏 JSON、错误结构、保存失败测试。
- 重复 `login_name` 校验测试。
- 导入覆盖规则测试。

### ~~12. 缺少 `.gitignore`~~

处理状态：已新增 `.gitignore`。

当前项目目录中有 `.idea/`、`__pycache__/` 和本地数据文件。

建议忽略：

```gitignore
__pycache__/
*.pyc
.idea/
data/accounts.json
data/*.invalid-*.json
```

是否忽略 `data/settings.json` 可以按需求决定。如果设置里只保存 Steam 路径，也建议忽略。

### 13. 非 Windows 环境缺少友好提示

项目定位是 Windows 本地工具，但 `system_utils.py` 顶层导入 `winreg`，非 Windows 会直接导入失败。

建议：

- 启动时检测平台。
- 如果不是 Windows，给出明确提示：“当前工具仅支持 Windows”。

### 14. 密码显示与剪贴板体验可以更谨慎

当前密码可以直接显示。

建议：

- 显示密码后几秒自动隐藏。
- 如果后续增加“复制密码”按钮，复制后定时清空剪贴板。
- 可以增加设置项控制是否允许明文显示密码。

## 建议实现顺序

1. 新增 `.gitignore`，先避免敏感数据和缓存文件误提交。
2. 实现 JSON 原子保存和加载结构校验，降低数据损坏和启动崩溃风险。
3. 限制 Steam 路径选择，防止账号密码传给非 Steam 程序。
4. 优化登录流程：关闭 Steam 前确认、登录结果文案更准确、失败提示更细。
5. 优化导入覆盖规则，避免清空已有字段。
6. 增加导入和仓库层测试。
